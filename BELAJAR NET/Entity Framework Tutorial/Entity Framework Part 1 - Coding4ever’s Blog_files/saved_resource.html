            Ya benar sekali om, Entity Framework memang lebih lambat dibandingkan menggunakan ADO.NET secara langsung.

            Tetapi bagaimana mungkin kita bisa meningkatkan produktivitas, ketika harus menulis kode seperti ini berulang-ulang :

            var daftarSupplier = new List<supplier>();

using (var conn = GetOpenConnection())
{
    var sql = @"SELECT SupplierID, NamaSupplier, Jalan, Kota
                FROM Supplier
                WHERE NamaSupplier LIKE @1
                ORDER BY NamaSupplier";

    using (var cmd = new SqlCommand(sql, conn))
    {
        cmd.Parameters.AddWithValue("@1", "%" + namaSupplier + "%");

        using (var dtr = cmd.ExecuteReader())
        {
            while (dtr.Read())
            {
                var supplier = new Supplier
                {
                    SupplierID = dtr["SupplierID"] is DBNull ? 0 : (int)dtr["SupplierID"],
                    NamaSupplier = dtr["NamaSupplier"] is DBNull ? string.Empty : (string)dtr["NamaSupplier"],
                    Jalan = dtr["Jalan"] is DBNull ? string.Empty : (string)dtr["Jalan"],
                    Kota = dtr["Kota"] is DBNull ? string.Empty : (string)dtr["Kota"]
                };

                daftarSupplier.Add(supplier);
            }
        }
    }
}

            Sementara menggunakan EF (Entity Framework), kita cukup menulis seperti ini :

            var daftarSupplier = new List<supplier>();

using (var db = new RetailContext())
{
    daftarSupplier = db.Suppliers
                       .Where(s => s.NamaSupplier.Contains(namaSupplier))
                       .OrderBy(s => s.NamaSupplier)
                       .ToList();
}

            Jadi harus adil donk om ketika membandingkan EF dan ADO.NET, jangan hanya dari satu sisi :D

            Yang satu lebih lambat dari sisi kinerja sistem, yang satu lagi ketinggalan jauh dari sisi penulisan kode. Jadi siapa yang lebih lambat ? :D

            Jaman sekarang performance bukan lagi faktor yang harus dipertimbangkan dalam proses pemilihan framework ORM seperti Entity Framework. Kalau aplikasi yang kita buat ternyata lemot, kemungkinan hal-hal dibawah ini yg menjadi tersangkanya:

            1. Teknik coding yang salah (biasanya disebabkan karena tidak menguasai dasar-dasar pemrograman seperti misalnya melakukan select * from tabel_transaksi yang berisi jutaan record)
            2. Programmernya tidak tahu cara pakai framework/teknologi tersebut
            3. Teknologi tidak sesuai peruntukannya, jadi harus disesuaikan kondisinya kapan harus pake ORM, saya sendiri minimal menggunakan Dapper.NET dan ekstensinya Dapper Extensions, jadi tidak pernah lagi menggunakan library ADO.NET secara langsung
            4. Belum mengetahui / menerapkan tips dan trik tuning performance untuk teknologi yang dimaksud

            Logikanya saja jutaan orang sudah menggunakan Entity Framework, bahkan Microsoft juga merekomendasikan penggunaan Entity Framework sebagai teknologi akses data untuk aplikasi-aplikasi terbaru.

            Jadi, kalau ada masalah performance, tersangka utama bukanlah teknologinya, tapi usernya :P
